### 1. HarnessEngineering：AI 能在真正"出事会炸"的后端系统里写代码吗？

**作者**: 腾讯程序员（lancelotluo） | **来源**: 微信公众号

**核心实体**:
- **腾讯 CDN LEGO** — 腾讯 CDN 核心接入层系统，百万行核心代码，C++ 超大规模分布式系统
- **Nonstop** — Rust 零人工代码代理框架，20 天 AI 开发完成
- **Claude / Codex / Gemini** — 多模型对抗式 CR 中使用的三个模型

**核心概念**:
- **Harness Engineering 五层架构** — 围绕"上下文、约束、反馈"三大要素的完整闭环系统
  - Layer 1: 权限安全基座
  - Layer 2: 代码规则即编译器
  - Layer 3: 流程约束（功能→测试→CR 严格阻塞）
  - Layer 4: 上下文建设（Agent.md 项目宪法、安全纪律、领域知识、专业 Skill）
  - Layer 5: 反馈闭环（踩坑日志→规则→Skill→A/B 验证）
- **多模型对抗式 Code Review** — 三模型（Claude+Codex+Gemini）并行独立审查 + 交叉验证 + 自动收敛
- **AI Coding 问题根因** — 不会说"我不知道"、幻觉、改不全、模式匹配代替验证、缺乏环境意识
- **踩坑→规则→Skill 进化闭环** — PIT-001 级别的真实 bug 转化为持久化知识资产
- **角色演变** — 初级开发→AI 操作员；高级开发→Harness 工程师；架构师→人机协作架构师

**建议操作**:

| **更新** | `agent-skill-specification` | LEGO 有 31 个 Skill、34 条踩坑规则，是 Skill 在超大规模工程中的实际应用案例 |

### 2. 【万字】抛开 RAG 谈蒸馏.skill，大概率是形式主义

**作者**: 叶小钗 | **来源**: 微信公众号

**核心实体**:
- **OpenClaw** — 被提及为推动 Agent 进展的框架
- **Hermes Agent** — 被提及已有检索后整理/综合的影子能力
- **Cursor / Trae / Qoder** — AI 编辑器，内部使用 RAG
- **Coze / Dify / FastGPT / RAGFlow** — RAG 开发平台
- **DeepSeek API / qwen3-rerank / text-embedding-v4** — RAG 技术栈组件

**核心概念**:
- **蒸馏.skill 的本质是 Workflow 迁移** — Skill 里装的是工作流（SOP），不是完整能力
- **RAG 范式演进** — Naive RAG → Advanced RAG → Modular RAG → Agentic RAG
- **Skills + RAG 最小闭环** — Skill 负责流程编排，RAG 负责动态知识检索，模型负责推理衔接
- **RAG 是给予模型 CoT 的存在** — 不只是"检索资料给模型看"，而是让模型推理时有外部知识展开
- **查询改写 / 混合检索 / 结果重排** — RAG 三大优化策略
- **基于证据的推理（Evidence-based Reasoning）** — 模型对 RAG 召回结果做局部推理、冲突判断
- **AI 原生团队** — 一号位先用 AI，人负责"起心动念"和补足模型缺陷

**建议操作**:
| 操作 | 目标 | 理由 |
|------|------|------|
| **新建** | `concepts/agent/skills-rag-integration` | Skills + RAG 的整合是 Agent 工程的重要新范式，本文给出了架构设计和实操案例 |
| **新建** | `concepts/principles/rag-architecture` | RAG 范式演进（Naive→Advanced→Modular→Agentic）及优化策略的系统总结 |
| **更新** | `agent-skill-specification` | 补充"Skill 缺乏知识支撑时只是形式主义"的批判性视角，以及 Skills + RAG 整合方案 |
| **更新** | `openclaw` | 被提及为推动 Agent 进展的框架 |
