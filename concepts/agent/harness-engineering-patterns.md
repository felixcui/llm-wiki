# Harness Engineering 核心模式

> 本文是 [[harness-engineering-deep-dive]] 的模式补充，聚焦六种核心工程模式。

## 六种核心模式

2026 年 4 月，张碧泉系统梳理了 Harness Engineering 的六种核心模式，这些模式通过结构化上下文、任务拆解和记忆优化来提升 AI 智能体的处理质量和可控性。^[raw/articles/2026-04-30_深入浅出HarnessEngineerring之核心模式与理念.md]

> **来源**: 张碧泉《深入浅出Harness Engineerring之核心模式与理念》，2026-04-30

### 1. 持久化指令文件

最基础的模式——将规则、约束和流程写入持久文件（如 CLAUDE.md、CLAUDE_GLOBAL.md 等），避免每次对话从头开始。[[claude-code]] 和 [[hermes-agent]] 均采用此模式。代价是文件需随项目更新维护，否则可能误导智能体。

### 2. 作用域上下文组装

将指令按范围（组织级、项目级、任务级）拆分，让智能体能动态加载最相关的规则。代价是规则分散在多个文件，可读性变差，且不同范围规则可能冲突。

### 3. 分层记忆

将记忆分为三层：常驻的精华摘要、按需加载的细节、仅支持搜索的完整历史，以节省 Token。代价是实现更复杂，需设计信息如何分层、流动，并确保索引与实际数据同步。此模式与 [[harness-engineering-implementations]] 中讨论的 Git + SQLite FTS5 记忆架构一脉相承。

### 4. 做梦整理

定期在后台对记忆进行去重、清理和重组，类似"垃圾回收"，以保持记忆整洁有效。代价是整理本身消耗资源，且可能误删有用信息。

### 5. 渐进式上下文压缩

新的对话保留细节，稍旧的做轻量总结，更早的压缩成简短摘要，适合长对话任务。代价是压缩必有信息损失，后续需要细节时，智能体可能会"编造"。

### 6. 工作流与编排

核心是"分离"——通过拆解任务流程来提升复杂任务的处理质量和可控性。包含三种子模式：

- **探索-规划-行动循环**：严格分为三步（只读探索 → 与用户对齐的规划 → 拥有写权限的执行），避免盲目操作。代价是流程更慢，小任务显得笨重。
- **上下文隔离子智能体**：为不同阶段创建拥有独立上下文和权限的子智能体，防止信息相互污染。代价是需要额外协调信息传递。
- **分支-合并并行**：将可并行的子任务分发给多个在独立环境中工作的子智能体，最后合并结果。代价是合并更复杂，处理代码冲突难度增加。

### 辅助模式

文章还涉及了工具与权限层面的模式（渐进式工具扩展、命令风险分类、单用途工具设计）和自动化模式（确定性生命周期钩子），这些与 [[hermes-agent]] 的生命周期 Hook 机制、[[claude-code]] 的工具安全设计一脉相承。六种模式并非互斥，实际系统通常组合使用多种。每种模式均有维护成本、信息损失或流程变慢等代价，需要在效率与灵活性之间权衡。
