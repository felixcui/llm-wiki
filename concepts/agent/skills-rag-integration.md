# Skills + RAG 整合

将 [[agent-skill-specification|Skill]] 的流程编排能力与 RAG 的动态知识检索能力结合，让 Agent 既能按规范执行流程，又能基于实时、私有的知识做出判断。这是当前 Agent 工程中一个重要的新范式。

## 核心问题

纯 Skill 的局限在于：**Skill 只能教 AI 怎么做，却不一定能让 AI 知道为什么这么做**。^[raw/articles/2026-04-23_【万字】抛开RAG谈蒸馏.skill，大概率是形式主义.md]

社区流行的"蒸馏同事.skill"——将一个人的能力封装为 Skill——存在根本性问题：
- 蒸馏出来的往往不是完整能力，而只是最容易被整理、最容易被复用、也最容易被自动化的那部分——更接近 Workflow 而非"能力"
- 真正难蒸馏的是：为什么优先看这个文档、为什么知道这个问题要参考上个月的事故复盘、为什么知道这里不能照常规流程走——这些都和数据有关
- 没有高质量数据与知识支撑的 Skill，只是形式主义

## 整合范式

正确的组合方式是三者分工：^[raw/articles/2026-04-23_【万字】抛开RAG谈蒸馏.skill，大概率是形式主义.md]

| 组件 | 职责 | 类比 |
|------|------|------|
| **Skill** | 定义怎么做（任务编排） | 骨架流程 |
| **RAG** | 提供做这件事时所需的知识 | 动态知识支撑 |
| **模型** | 基于流程和知识把事情串起来 | 局部推理 |

没有 RAG 的 Skill 更像在模仿一个人的**动作**；有了 RAG 之后，Skill 才开始有机会模仿一个人做判断时的**信息组织方式**。

## 架构设计

### 骨架流程定义

Skill 负责定义骨架流程，不灌大量知识：
1. 识别用户问题类型
2. 判断是否需要查询知识库
3. 如果需要，构造检索问题
4. 从指定知识库中召回相关资料
5. 对召回结果进行过滤与重排
6. 基于结果回答用户
7. 知识不足时进入人工兜底

### 关键节点接入 RAG

凡是涉及"需要依据资料做判断"的地方，都通过 RAG 动态取知识，而非写死在 prompt 里。Skill 只负责决定**什么时候查、查什么**，RAG 负责把相关知识拿回来。^[raw/articles/2026-04-23_【万字】抛开RAG谈蒸馏.skill，大概率是形式主义.md]

### 检索后局部推理

RAG 检索回来的内容并不天然就能直接用，模型需要做局部推理：
- 判断哪些资料与当前问题最相关
- 判断多个资料之间是否冲突
- 最终基于可信的资料，形成相对稳妥的回答

这一步类似于 [[hermes-agent]] 中检索后的 summarization、memory recall 里的 reflect/synthesis 能力——不只是"查到就塞提示词"，而是往"检索后整理、综合、再利用"的方向走。

## 与 RAG 范式演进的关系

Agentic RAG（将 LLM 的自主规划能力与 RAG 结合，自主判断是否需要检索）被认为是最适合蒸馏.skill 的技术范式。现阶段 Skills + RAG 的最小闭环可视为 Agentic RAG 在 Skill 架构中的具体实现。^[raw/articles/2026-04-23_【万字】抛开RAG谈蒸馏.skill，大概率是形式主义.md]

## 多路召回

复杂任务的 Skill 不只查一个知识库，可设计并行多路召回（parallel_rag），从多个知识库分别检索后用 RRF（Reciprocal Rank Fusion）合并去重。这种设计已接近 Modular RAG 的思路。
