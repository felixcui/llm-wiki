# Token ROI（Token 投资回报率）

> Token 消耗量是成本，不是成果。衡量 AI 使用效果的真正标准是 token ROI——即 AI 带来的有效收益与综合成本之比。^[raw/articles/2026-05-06-token-roi.md]

由曹犟在《别晒 token 了，先算算 token ROI》中提出。核心论点：当前社区热衷于晒 token 消耗量并将其视为 AI 使用深度的标志，但这本质上是用容易量化的过程指标替代了创造价值的结果指标。

## 为什么 token 消耗量具有误导性

Token 消耗量之所以容易被追捧，是因为它**好统计、好比较**，能给人"我也参与进来了"的感觉。但高消耗可能来自多种截然不同的情况：

- **高价值探索**：解决复杂问题，投入本来就该高
- **低效重复**：token 花在无效沟通和反复返工上
- **低价值任务的"AI 化"**：表面热闹，实际没有实质产出

这与管理学中经典的古德哈特定律一致——**当一个指标成为目标，它就不再是一个好指标**。过去组织也犯过类似错误：比较代码行数、会议次数、加班时长，现在变成了比较 token 消耗量。

## Token ROI 的定义

```
Token ROI = AI 带来的有效收益 / AI 使用的综合成本
```

**收益维度**：
- 缩短调研时间（从半天到半小时出初稿）
- 提高交付物质量（更多版本比较、更少的边界条件遗漏）
- 降低试错成本（原来因成本太高被跳过的环节得以补回）
- 做过去做不了的事（系统整理用户反馈、快速理解陌生代码库）

**成本维度**：
- Token 本身的计费成本
- 编写 Prompt、整理上下文的人力投入
- 验证结果、维护 harness 的工程成本
- AI 出错后的返工成本（参见 [[comprehension-debt]]）

## 判断 Token ROI 的三个信号

1. **工作流是否真正改变**——不是把搜索引擎换成聊天框，而是工作中的某些环节因 AI 被重新设计（串行变并行、补回被跳过的高成本环节）
2. **结果是否变好**——交付物的质量、速度、完整性是否有可感知的提升
3. **是否开始做过去做不了的事**——从"想想算了"变成"可以试试"

## 与云计算成本优化的类比

Token 消耗与云计算资源消耗存在高度平行：云账单很高可能源于业务规模增长（好事），也可能源于架构设计不佳（坏事）。[[agent-technical-debt]] 中提到的**成本治理**模块正是这一问题的 Agent 侧表现——Agent 可能陷入重试循环持续消耗 token，需要按 Agent/团队/用例进行成本分解。^[raw/articles/2026-05-06-token-roi.md]

## 企业视角：三本账

组织应避免简单搞 token 消耗排行榜（考核指标会导致行为扭曲），转而关注：

1. **场景账**：哪些高价值、高频、可验证的场景已开始用 AI？
2. **结果账**：节省了多少人天、减少了多少返工、提升了多少转化？
3. **沉淀账**：哪些经验被固化成模板、脚本、知识库或工作流，而非每次重新对话？

沉淀账与 [[prompt-engineering]] 中的 Structured Outputs 和 Few-shot 模式形成呼应——将有效经验固化为可复用的结构，本身就是提升 token ROI 的手段。

## 核心原则

> **不是要少用 token，而是要提高 token 的投资回报率。**^[raw/articles/2026-05-06-token-roi.md]

Token 花出去只是成本，换回来的工作流变化、结果改善和能力提升，才是真正的回报。
